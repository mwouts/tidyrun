# TidyRun Development Guidelines

## Environment Setup

This project uses **pixi** for environment management. The main development environment is configured in `pyproject.toml` with support for multiple Python versions (3.9-3.14).

### Pixi Commands
- **Install**: `pixi install`
- **Run tests**: `pixi run pytest`
- **Run specific environment**: `pixi run -e [env-name]`
- **Available environments**: default, build, docs

### Key Dependencies
- **toml**: TOML parsing
- **pandas**: Data manipulation
- **pyarrow**: Parquet support
- **pytables**: HDF5 support
- **pyright**: Type checking (dev)
- **pre-commit**: Git hooks (dev)

## Pre-commit Requirements

All commits must pass pre-commit hooks defined in `.pre-commit-config.yaml`.

### Setup
```bash
pre-commit install
```

### Hooks Applied
1. **Standard checks**: JSON validation, YAML validation, file ending fixes, trailing whitespace
2. **Ruff formatting**: Auto-fixes linting issues and formats code
3. **Pixi lock update**: Automatically updates `pixi.lock` when `pyproject.toml` changes

### Running Manually
```bash
pre-commit run --all-files
```

## Type Checking with Pyright

This project uses **Pyright** with strict type checking enabled for source code and documentation.

### Configuration
- **Type checking mode**: standard
- **Strict mode**: Applied to `docs/` and `src/tidyrun/`
- **Minimum Python**: 3.9

### Validation
```bash
pixi run pyright
```

All new code in `src/tidyrun/` must pass strict Pyright checks. Type annotations are required.

## Code Quality Standards

### Documentation Consistency
- **Documentation must be updated alongside code changes**
- Keep docstrings, README, and `docs/` files in sync with implementation
- Update examples if function signatures or behavior changes
- API documentation changes are part of the pull request, not separate

### No Silent Failures
- **Do NOT use broad try/except blocks** that silently catch and ignore exceptions
- All exceptions must be handled explicitly with clear intent
- Use specific exception types (e.g., `except ValueError:` not `except Exception:`)
- If an exception must be caught and ignored, add a clear comment explaining why

### Example - Good
```python
try:
    value = parse_int(input_str)
except ValueError:
    # Expected when input is not a valid integer
    value = 0
```

### Example - Bad
```python
try:
    value = parse_int(input_str)
except:  # Broad, silent failure
    pass
```

## Workflow Summary

1. **Make changes** respecting type hints and code style
2. **Run tests**: `pixi run pytest`
3. **Check types**: `pixi run pyright`
4. **Commit**: Pre-commit hooks will validate automatically (or run `pre-commit run --all-files`)
5. **Push**: All checks must pass before merging
